Triyo file collaboration

Centralizing task management processes for assignees

Team

Designer (yours truly) and product manager

Timeline

March 2021 - April 2021

Background

Context

Triyo is a Toronto-based startup that builds project and task management software catered to users in the finance industry. By integrating the Microsoft Suite into their software, Triyo allows project teams to collaborate on files without excessive change management. A file collaboration task is created when project managers assign a section of a Word document to team members to edit or approve.

There are 7 steps assignees use for file collaboration tasks (A - G)

1. Understand task requirements and gain context

A

Read comments from product manager for task instructions

B

Open and read supporting documents attached to comments

C

Read full original document for context

D

View assigned section of document

2. Edit assigned section

E

Edit the assigned section online

F

Send the edits for review

Because of the new Word add-in, all edits will now exist offline. Steps 5 and 6 will no longer exist in the web application.

3. Ask questions and communicate with project manager

G

Leave a question or comment

Business goals

Currently, project managers upload Word documents to the Triyo web application and assign sections that are edited online.

Recently, Triyo launched a Word add-in, which means assignees can edit their sections offline.

The code used support the online editor was expensive and difficult to maintain.

Stakeholders wanted to direct users to the new Word add-in for offline edits and remove online editing from the application.

With the new business goals in mind, how might this affect the 7 steps of the file collaboration process for assignees?

Problem definition

Problem statement

Redirecting users to offline editing diminished the web application’s role in the user journey. The assignee will primarly use the web application for steps A, B, C, D, and G.

These steps must be optimized for assignees so they can gain context and communicate with their project manager before moving offline for edits.

Current state heuristic analysis

Tasks are listed in rows on a table. The assignee clicks on action icons in the row to view dropdown selections for task details.

User journey

The workflow demonstrates how assignees navigate the current state to perform steps A, B, C, D, and G.

Pain points

There are two major pain points with this user journey.

Repetitive actions

Too many modals disrupt the user journey and lead to repetitive back-and-forth actions.

Poor accessibility to information

Information is hidden in dropdowns or in-between modals.

Design goals

Reduce repetitive actions so users can begin editing in an efficient manner.

Improve access to important information so users have a better understanding of their task.

Support scalability of the product as new features are added.

Solution

Dedicated tasks page

A dedicated task page improves access by centralizing task information.

Tabular navigation

Tabs replace modals to reduce repetitive actions when navigating between screens while providing room for scalability and new features.

Feature no. 1

Audit trail comments are used by task managers and assignees to communicate with one another

Initial iterations

Upon creating a dedicated task file collaboration page, I divided the screen so the Audit Trail would always be visible. This is important because the Audit Trail records all activity associated with the task and is where communication between team members occurs. My assumption was that most comments in the Audit Trail would be brief updates with a few lines of text.

Final iterations

The director of product explained Audit Trail comments are generally longer and can be several paragraphs. This is because Triyo is integrated with Outlook, so project managers will send task requirements over email. The final design truncates the comment like Outlook would.

Feature no. 2

Each file collaboration task has 4 file states: The original file, the assigned section of the file, the edited section of the file, and the updated file

Initial iterations

There were four states of the document to accomodate for. Using tabs to navigate between states was much more efficient that clicking in and out of modals. However, I received feedback that this navigational structure gave the appearance that the state of the document was associated with the activity described in the selected audit trail comment, which was not the case. The hierarchy of information was skewed.

Final iterations

The audit trail is separated from the document states. The icons are replaced with text, which requires less cognitive load for the user.

Testing

The prototype was given to three new hires within the company for feedback.

All three users commented that the new prototype was either more centralized or more efficient than the current design.

The audit trail was present between both tabs for convenience, but this was confusing for users. The transition from left to right sides of the screen was jarring.

Conclusion

Recommended future Steps

  • Iterate on user feedback

  • Prepare files for handoff and support developers in build

Learnings

We can align business goals with the user journey

Business goals were the inception for this project, but they can always be aligned with a better user experience. For this project, it was important to frame business goals with the user journey. Doing so opened up an opportunity for better UX within the product.

How to initiate better conversations in a fast-paced environment

Working in a startup environment meant two-week sprints to research, define, and ideate. It is intimidating to push the envelope when time is of the essence! I found demoing multiple design options to stakeholders often led to more engaging conversations. I learned to prototype quickly and be less restrictive in initial stages with my low fidelity design options in order to encourage these types of user-centered discussions during early morning design reviews.
