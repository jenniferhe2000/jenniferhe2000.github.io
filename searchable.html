During a 2 week sprint at Loblaw Digital, I defined searchable select menu interactions for our internal Campaign Manager, reducing production time and enabling the production of personalized campaigns at scale. In one year, the difference in revenue between personalized campaigns built in Campaign Manager and non-personalized campaigns was +5.8M.

My role

In January 2024, I joined the personalization team at Loblaw Digital. Our team was responsible for our internal Campaign Manager, a campaign management tool used by our colleagues to produce personalized campaigns for 1M+ customers across Canada. When I joined, Campaign Manager had just launched the previous month.

Problem

  • Because unique content must be created for every audience segment, campaign personalization requires more content. Menus on our interface connected to our content databases were growing, and items became more and more difficult for colleagues to find.

  • Personalized campaign production is repetitive – a unique campaign is created for every audience segment – so our colleagues interfaced with these long menus on a day-to-day. Colleagues reported frustration because they had to repeatedly find items buried within our long menus.

  • In order for personalized campaign production to scale, even more content needs to be created, which means menus will only get longer.

Solution

Why was this an effective search solution for us?

  • A searchable select menu was a lower effort alternative to typeahead search

  • Task-analysis validated that our colleagues only need to filter our menus after typing a keyword, but providing additional search suggestions was not necessary

Defining searchable select menu interactions

Using best practices and speaking to our colleagues

The height of the select menu is flexible

  • I learned that ideally, colleagues want to see all menu items immediately on focus: ”If I immediately see the [item] I need to find in the menu, I can just click on it there”

  • If it is not possible to show all menu items, the menu can be set to a max. height for each use case

Menu items must be sorted alphabetically

The current experience listed items in alphabetical order, and this was validated by our colleagues to be a good experience – The [items] are listed in alphabetical order, which is helpful in finding what we need right now”“

Menu items must be parsable from anywhere

Menu items in our internal systems had legacy names with prefixes that were unintuitive. Our colleagues should be able to ignore prefixes upon search.

Search cannot be case-sensitive

Furthermore, menu items in our internal systems had legacy names formatted in camelCase. Our colleagues should not have to remember which characters in the target item are upper or lowercase.

Bold substrings of the result that match the user input

This is a best practice that provides the user with visual feedback of their inputs.

Learnings

Reflections from this 2-week sprint

There is more to search than slapping an input field and menu together

The searchable select menu reused existing input field and menu components, but I had to consider how these two components would speak to each another. What is the height of the menu? How should we handle case-sensitivity? Solutions were found by thinking about how our colleagues work and validated as feasible by the tech team.

A activity like this is not isolated from user-centred design

It was less intuitive to foresee how user research had a role in this activity. However, task analysis was essential to validating the search solution was effective.
